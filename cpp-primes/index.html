SUMMARY

This program determines all prime numbers less than 10,000. It does this by spawning 25 threads running in a Linux process environment. There are 25 primes less than 100. For each prime less than 100, there is a distinct thread. Each thread determines which numbers from 2 to 10,000 are multiples of this prime. For example, the thread for prime 5 will determine that 5, 10, 15, 20 ... are multiples of 5 and thus these numbers are not prime numbers.

There is a shared resource for collecting the set of all non-prime numbers. The 25 threads are competing to update this shared resource. This shared resource is a C++ Standard Template Library "set" container. This is a race condition. The C++ <thread> library is used to protect the shared resource by means of its mutex capability.

PURPOSE

This program demonstrates how to overcome several issues related to real-time, multi-thread processes:

  • Race Conditions
  • Thread Deadlocks
  • Functions That Are Not Thread Safe

RUNNING THE PROGRAM

Because there are many threads, the order that the threads start and finish is not deterministic. The final result is a list of all of the primes less than 10,000. There are 1229.

OVERCOMING RACE CONDITIONS

The threads that compute the non-primes use the C++ <thread> library to protect the shared resource with a mutex. This is shown in the code surrounded by the comments "// PROTECTED REGION" and "// END PROTECTED REGION". There is a mutex lock then a write to the resource then a mutex unlock: Code that has a protected region

This protected region is very safe:

  • The protected code segment is very short
  • Only one shared resource is modified
  • There is no branching nor function calls within the protected region
  • The only data modified is the shared resource
  • There is no nesting of mutexes

OVERCOMING THREAD DEADLOCKS

The thread that determines final results is waiting for a signal from a condition variable indicating that all other threads have finished updating the set of non-primes. A deadlock will occur if the signal is not received. To force this deadlock, modify the number of finished threads from 25 to 26 at this line of code in Threads.cpp: 

const unsigned numberThreads{25}; // to force a deadlock change 25 to 26
When the program is run, the 25 threads all run to completion but then the process hangs, waiting forever for a 26th thread to complete. To correct this deadlock condition, modify the wait() so that the lock times out after 10 seconds: 
/*
 * Wait for the condition variable signal
 */
std::unique_lock ul(SharedResources::mutexForConditionVariable);
SharedResources::conditionVariable.wait(ul);

        change the wait to be:

SharedResources::conditionVariable.wait_for(ul, 10s);
When the program is run again, the 25 threads all run to completion, then there is a pause for 10 seconds, then the thread continues and correctly determines the 1229 prime numbers.

THREAD SAFE FUNCTIONS

A C++ function or method will be thread safe as long as all of the referenced data is on the stack. All data local to the function must not be static. There must be no references to global data. However, there may be references to shared resources as long as they are protected by mutex lock/unlock. This function in Threads.cpp is thread safe: 

std::set<unsigned> ThreadPrimes::universalSet()
{
  /*
   * Create the universal set, a set of numbers from 2 to 10,000 inclusive
   */
  std::set<unsigned> universal; 
  for(size_t i = 2; i <= maxNumber; i++)
  {
    universal.insert(i);
  }
  return universal;
}
